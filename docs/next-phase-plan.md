# 下一阶段计划：Atlas 工作台到 3D 回灌

更新时间：2026-05-01

## 阶段目标

把当前原型从“能在 3D 里跑起来”推进到“能稳定策划、校正、解释一块真实地理区域”。下一阶段的核心不是继续堆视觉元素，而是让二维地貌总览图成为策划工作台，再由它驱动 3D 表现和玩家行动。

## P0：文档与工程整编

状态：本轮已开始。

验收标准：

- 有一份文档入口：`docs/README.md`。
- 有一份上下文压缩摘要：`docs/context-summary.md`。
- README 不再把当前真实 DEM 误描述为程序化占位母版。
- 小范围代码整理有测试保护，不破坏当前构建。

## P1：2D Atlas 工作台

状态：已完成第一刀，已有全屏 Atlas Workbench、图层开关和点击信息卡。

目标：先让人能在平面图上读懂区域。

任务：

- 增加 Atlas 图层开关：地貌、水系、古道、城市、关隘、军事、民生、人文。`已完成第一版`
- 增加 feature hover/click 信息卡：标题、类型、历史/地理解释、3D 反馈规则。`已完成 click 第一版`
- 增加全屏 Atlas Workbench：让地图可作为游戏导航与开发校对界面。`已完成第一版`
- 增加标签优先级和基础避让，避免默认视图文字糊在一起。
- 增加“总览/水系/道路/军事/民生/人文”专题模式。

验收标准：

- 打开地貌总览后，不看 3D 也能解释关中、秦岭、汉中、剑门、成都平原的关系。
- 用户能按需展开信息，而不是一上来被所有标记淹没。
- Atlas feature 仍然全部使用严格世界坐标，不引入单独的 UI 坐标体系。

## P2：水系与地貌造型规则

状态：水系已提升为独立现代 hydrography 管线优先级。

目标：让水系和地貌粒度真正影响视觉，而不是只画几条线。

任务：

- 建立现代水系事实层：名称、等级、流域、来源、可信度、关系字段。
- 为秦岭区域生成独立 hydrography asset，不再只依赖 `qinlingAtlas.js` 手工折线。
- 建立 L0/L1/L2 水系显示规则，全国/区域/局部不同密度。
- 加入 DEM 校验，报告河流是否贴近低地、谷地、盆地和合理流向。
- 把渭河、汉水、嘉陵江、褒水、斜水接入 terrain coloring。
- 为河谷、盆地、平原、主脊、峡谷建立不同造型粒度。
- 平原和盆地做低频平滑，山脊保留中高频，关隘/峡谷局部强化。
- 在 DEM 生成或运行时增加 feature-aware smoothing / emphasis 的实验分支。

验收标准：

- 关中平原、汉中盆地、成都平原在 3D 中能被直接看出来。
- 不再出现“全图同一粒度尖峰”的视觉问题。
- 水系能帮助读出河谷、盆地和路线，而不是装饰线。
- 现代水系优先做准；历史河道和古湖泽只作为未来 `Era Overlay` 预留。

## P3：道路、移动与叙事反馈

目标：让“走路”成为理解地理的方式。

任务：

- 让 route affinity 影响速度、体力/阻力、镜头提示和地面提示。
- 离开古道进入陡坡时，移动速度下降且视觉上给出地形风险提示。
- 对陈仓道、褒斜道、傥骆道、子午道、剑门蜀道分别设置路线性格。
- 主线任务从“到点触发”升级为“沿路线理解空间结构”。

验收标准：

- 玩家能凭移动手感感受到平原、山道、关口、盆地之间的差异。
- 主要古道不只是地图线，而是行动策略。

## P4：开放世界骨架

目标：让当前切片能扩展到全国。

任务：

- 增加 `world manifest`，描述全国低精度母版和重点 region 入口。
- 增加 `region registry`，支持多个区域 manifest。
- 设计跨区域加载和状态保持策略。
- 按“重点区/标准区/稀疏区/极稀疏区”配置速度、镜头、细节密度、事件密度。
- 增加多尺度地图契约：当前秦岭切片作为 `Overworld Atlas`，未来通过 `Gate Entrance` 进入 `Local Scene`。
- 增加历史年代层契约：近期以现代 DEM 为基础，未来用 `Era Overlay` 表达河道、湖泽、关隘和城市意义变化。

验收标准：

- 当前秦岭 region 不再是唯一硬编码世界。
- 可以注册第二个 region 而不重写主运行时。
- 可以在不填充历史内容的情况下，说明时代层如何挂接到现代基础地图上。

## P5：性能与可用性

目标：浏览器里能持续跑，而不是只在空场景里看起来能跑。

任务：

- 增加 runtime stats 面板或开发模式指标。
- 跟踪 terrain mesh、chunk 数、scenery 数、draw calls、内存趋势。
- 重点区域采用更严格预算：更多内容但不爆对象数。
- 为低性能设备准备 Atlas-only 或远景优先模式。

验收标准：

- 重点区域内容变多后仍能保持稳定帧率。
- 性能问题能被定位到 chunk、scenery、水系、标签或材质层。

## 推荐执行顺序

1. 先做 P1，因为它会决定 POI、水系、道路和文案的事实源。
2. 然后做 P2，把 2D 的地貌规则回灌到 3D。
3. 再做 P3，让行动体验绑定地理结构。
4. P4 和 P5 可以穿插推进，但不应该抢在 P1/P2 之前大做。
