# 山河中国 · 首版技术方案

## 目标

首版不是做一张全国历史大沙盘，而是做一个可在浏览器打开的 3D 垂直切片：

- 区域：`秦岭 - 关中 - 汉中 - 四川盆地`
- 体验：骑马漫游、低提示、强氛围、轻收集
- 理解：让玩家自然感到山脉、盆地、河谷、关隘如何塑造生活与战争

## 首版原则

1. `DEM 作为母版`
   地形关系由真实高程数据提供，游戏不直接照搬原始 DEM，而是在此基础上做降采样、平滑与夸张。
2. `镜头可旋转但不自由飞`
   维持塞尔达式斜俯视沉浸感，同时允许玩家绕角色 360 度观察山势。
3. `知识不是弹窗灌输`
   玩家只拾取一句短提示，完整知识进入收藏册，避免破坏空灵感。
4. `场景先成立`
   风、云、光、雾、远山比重型 HUD 更重要。

## 技术栈

- `Vite`
- `TypeScript`
- `Three.js`

选择原因：

- 浏览器端部署简单，适合快速迭代原型。
- Three.js 对高度场地形、轻交互、后期扩展都比较稳。
- 首版不急着引入大型游戏引擎，先把玩法语言和视觉气质立住。

## 地形数据方案

### Runtime 结构

运行时需要的不是原始 GeoTIFF，而是一份轻量的切片资产：

- `bounds`: 原始地理范围
- `resolution`: 网格尺寸
- `heights`: 归一化高程数组
- `riverMask`: 重要河谷/水系掩码
- `passMask`: 关键通道/关口掩码
- `settlementMask`: 适合聚落的区域热度

当前仓库已经采用这套结构，运行时读取：

- `public/data/qinling-slice-dem.json`

当前这份文件仍是“程序化原型母版导出物”，但运行时接口已经和后续真实 DEM 接入方式对齐。

建议导出格式二选一：

1. `heightmap PNG + metadata JSON`
2. `compressed JSON`

首版更推荐 `PNG + JSON`，体积和加载都更友好。

当前临时实现仍使用单个 JSON，原因是：

- 先把 DEM 资产边界与运行时采样逻辑稳定下来
- 方便快速验证高度、河谷、山口、聚落热区等多个通道
- 等真实 DEM 接入后，再考虑压成 `PNG + metadata JSON`

### DEM 处理管线

目标流程：

1. 获取 DEM 源数据
   可选 `Copernicus DEM`、`SRTM` 或国内可公开使用的高程源。
2. 裁切区域
   先锁定秦岭至四川盆地一块，而不是全国。
3. 降采样
   把原始栅格压到适合网页实时渲染的分辨率，比如 `256 x 320`。
4. 平滑与去噪
   去掉首版不需要的小沟谷和噪点。
5. 设计夸张层
   对秦岭山脊、山口、汉中盆地、入蜀谷道、成都平原做定向强化。
6. 导出游戏资产
   让运行时只读取轻量结果。

当前仓库内已经有一条原型脚本管线：

- `npm run build:dem`
- 脚本文件：`scripts/generate-qinling-dem.mjs`

它目前会把程序化地形导出为 DEM 风格资产，用于验证：

- 高度场采样
- 掩码通道读取
- 场景运行时与资产解耦

下一步再把这条脚本替换为“真实 DEM 输入 -> 处理 -> 导出”。

### 为什么不直接在运行时加载原始 DEM

- 体积大
- 原始细节过碎，不利于游戏阅读
- 浏览器端实时处理成本不划算
- 设计师需要一个可以“故意不真实”的夸张层

## 地图抽象规则

必须保留：

- 秦岭的横向阻隔感
- 关中的开阔与腹地感
- 汉中的门轴感
- 入蜀道路的压缩与险峻
- 成都平原进入后的舒展与安全感

允许夸张：

- 山脊更挺
- 山口更窄
- 盆地更完整
- 城市和要道辨识度更高
- 真实距离适度压缩，保证 15 到 20 分钟能穿越全图

## 镜头与移动

### 镜头

- 默认斜俯视
- 支持水平 360 度旋转
- 俯仰角限制在小范围，避免自由飞破坏阅读
- 轻微缩放
- 镜头跟随平滑，强调漂浮感而不是硬追踪

### 移动

- 骑马第三人称移动
- 移动方向相对镜头
- 上坡减速，下坡稍快，山道显著更慢
- 进入关口和谷道时，让玩家自然感到“路被收窄”

## 视觉表现

关键词：

- `空灵`
- `微缩山河`
- `低模雕塑感`
- `中国山水气韵`
- `少 UI`

建议手段：

- 柔和雾效
- 分层远山
- 水面高光
- 风向粒子或流动纹理
- 低饱和大地色 + 温暖点金

## 知识收集系统

### 交互原则

- 不自动弹大段文字
- 玩家靠近触发物时，仅出现非常轻的感知提示
- 收集后只显示一句短文
- 完整说明进入收藏册

### 数据结构

每条知识碎片至少包含：

- `id`
- `title`
- `zone`
- `position`
- `pickupLine`
- `details.geo`
- `details.history`
- `details.strategy`

### 表现形式

世界内建议做成：

- 山河残简
- 古图残页
- 石刻回响

首版统一成一种即可，避免美术语言过散。

## 项目结构建议

```text
src/
  data/
    qinlingSlice.ts
    fragments.ts
  game/
    terrain.ts
    camera.ts
    journal.ts
  main.ts
```

## 实施顺序

1. 先把全国概念图收束成秦岭切片
2. 加入旋转镜头与更轻的 HUD
3. 做首批知识碎片收集与收藏册
4. 把地形函数重构为“母版层 + 夸张层”
5. 再接入真实 DEM 资产导入流程

## 当前代码阶段

当前仓库里的原型仍是“概念化全国地图”。下一步实现会先把它收束成切片原型，但在真实 DEM 资产尚未接入前，会暂用程序化母版结构来占位，同时把接口留好，便于后续切换成真实 DEM 数据。
