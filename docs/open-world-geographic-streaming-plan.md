# 山河中国 · 开放世界地理资产流式加载方案

## 1. 方案目标

未来如果 `山河中国` 是一个可以持续乱逛、并且逐步加入叙事主线的开放世界，
那么地理资产就不能继续采用：

- 单张整图一次性加载
- 所有区域同精度常驻
- 地形、mask、POI、叙事触发全部绑在一个文件里

这份方案的目标是定义一套适合浏览器的地理 streaming 架构，使系统能够同时满足：

- 全国尺度连续存在
- 重点区域高密度展开
- 第三人称视角下的远景与近景层次
- 浏览器端可承受的内存与加载成本

## 2. 核心结论

未来推荐采用：

`全国低精度常驻 + 区域高精度按需加载 + 重点区 chunk 流式加载 + 叙事资产独立加载`

也就是说，世界不再只有“一张地图”，而是多层同时存在：

1. `L0 全国母版`
2. `L1 区域切片`
3. `L2 区域 chunk`
4. `L3 POI / narrative / scenery`

## 3. 四层流式结构

## 3.1 L0 全国母版

作用：

- 提供全国连续地形骨架
- 让玩家无论在哪里都不会掉出世界
- 支撑远景与大尺度方向感

特点：

- 低精度
- 可常驻内存
- 覆盖全国
- 不承担重点区域细节

## 3.2 L1 区域切片

作用：

- 表达某一大区域的主地貌逻辑
- 作为重点区高精度内容的入口层

特点：

- 比全国母版精细
- 可按玩家所在大区加载
- 不一定一次载入所有重点区

## 3.3 L2 区域 chunk

作用：

- 将重点区域进一步切成按需加载的小块
- 控制第三人称视角下的局部内容密度

特点：

- 粒度小
- 可加载 / 卸载
- 与玩家位置强绑定

## 3.4 L3 附加语义与叙事层

作用：

- 承载不是单纯高程的数据
- 按需提高区域叙事和玩法密度

内容包括：

- POI
- 知识碎片
- 地标
- 叙事触发器
- 局部 scenery 标记
- 重点路径 / corridor 数据

## 4. 推荐的运行时加载逻辑

## 4.1 常驻内容

运行时应始终常驻：

- 当前天空系统
- 全局光照与天气框架
- `L0` 全国低精度地形
- 玩家附近的最小必要导航 / 定位信息

## 4.2 区域级加载

当玩家进入某个重点大区范围时：

- 加载该区域 `L1 manifest`
- 根据 manifest 确定 chunk 布局
- 准备该区域的高精度地形和语义层

## 4.3 chunk 级加载

根据玩家当前 `world position`：

- 激活玩家所在 chunk
- 同时预加载周围一圈 chunk
- 再保留少量外圈缓冲 chunk

建议至少使用三圈逻辑：

1. `active`
   玩家当前 chunk 及最近邻
2. `warm`
   即将进入的外围 chunk
3. `cold`
   尚未加载，但可快速判定

## 5. 推荐的加载半径模型

后续实际数值可以调，但设计上建议先采用：

### 5.1 地形 chunk

- 当前 chunk：必须加载
- 邻近 1 圈：常驻
- 邻近 2 圈：预热

### 5.2 植被 / 建筑 / scenery

- 只在玩家周围较小半径内加载高密度版本
- 中远处只保留抽象表达

### 5.3 POI / narrative

- 以 chunk 为单位预读
- 只在进入一定触发半径时激活交互逻辑

## 6. 卸载逻辑原则

流式世界不只是“加载”，也必须明确“什么时候卸载”。

推荐原则：

### 6.1 立即可卸载内容

- 已远离的近景草木实例
- 已远离的高密度 scenery
- 不在可见范围的局部建筑细节

### 6.2 延迟卸载内容

- 刚离开的 chunk 地形
- 相邻区域的预热数据

原因：

- 避免玩家转身或折返时频繁抖动加载

### 6.3 尽量常驻内容

- 全国母版
- 当前大区 manifest
- 少量全局 narrative state

## 7. 第三人称视角对 streaming 的特殊要求

第三人称视角比高俯视更容易暴露 streaming 问题，因为玩家会同时看到：

- 远山
- 天空
- 地平线
- 中景林带
- 近景草木和建筑

因此 streaming 必须做到：

1. 远景连续
2. 中景不突然空掉
3. 近景切换尽量不跳
4. LOD 和 chunk 边界尽量平滑

## 8. 推荐的视觉-加载分工

## 8.1 远景

来源：

- 全国母版
- 低成本区域大轮廓
- 雾、天空、云层

要求：

- 连续稳定
- 不依赖高精度 chunk

## 8.2 中景

来源：

- 区域切片
- 相邻 chunk 的低到中精度内容

要求：

- 保证空间关系可读
- 保证地形与林带关系不突兀

## 8.3 近景

来源：

- 当前 chunk 的高精度地形
- 当前 chunk 的植被 / 建筑 / POI / narrative

要求：

- 只在玩家周边有限范围内足够丰富

## 9. 推荐的 chunk 内容拆分

一个 chunk 不应该只有高程。

推荐分成几个并行可加载文件或数据段：

### 9.1 terrain

- heights
- min/max height
- world bounds

### 9.2 masks

- river
- pass
- settlement
- corridor

### 9.3 scenery

- tree clusters
- vegetation scatter seeds
- rock groups
- building anchors

### 9.4 poi

- landmarks
- fragments
- trigger zones

这种拆分的好处是：

- 地形和语义可以按不同优先级加载
- 加载失败时更容易降级
- 不必每次把所有内容一起读进来

## 10. manifest 的作用

流式架构里，manifest 是必须的。

至少需要：

### 10.1 world manifest

描述：

- 全国边界
- 全国 world size
- 可用区域列表

### 10.2 region manifest

描述：

- 区域边界
- 区域 world size
- 放大倍率
- chunk 布局
- chunk manifest 路径

### 10.3 chunk manifest

描述：

- 各 chunk 的索引
- chunk 对应世界边界
- chunk 对应现实边界
- 资源文件路径

## 11. 降级策略必须预先设计

浏览器 streaming 世界必须允许部分内容降级，而不是“不全就崩”。

推荐降级顺序：

1. 优先保证地形存在
2. 其次保证主要地标存在
3. 再保证植被与建筑丰富度
4. 最后才追求动物和动态细节

也就是说，最糟糕情况下也应做到：

- 世界连续
- 方向可读
- 地貌成立

而不是因为一层 scenery 没加载到就让整个区域不可玩。

## 12. 当前从原型迁移的推荐步骤

### 12.1 第一步

让当前单文件切片资产支持：

- `bounds`
- `world`
- `grid`
- `regionId`
- `lod`

### 12.2 第二步

增加 region manifest，
即便底层还只有单张切片文件，也先让运行时能“从 manifest 找到资产”。

### 12.3 第三步

把重点区拆出 chunk manifest，
哪怕第一版 chunk 还是粗粒度，也先建立机制。

### 12.4 第四步

把 fragments / landmarks / poi 改成 region 或 chunk 组织，而不是全局常量直挂。

## 13. 当前建议的一句话版

`山河中国` 的开放世界地理加载应采用“全国低精度骨架常驻、区域高精度按需进入、重点区 chunk 流式加载、语义与叙事层独立加载”的四层结构；只有这样，第三人称视角下的全国连续探索与重点区域高密度体验才能在浏览器里同时成立。`
